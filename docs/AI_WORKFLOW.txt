--------------------------------------------------------
 SECTION 1 — VERSIONING STANDARD (Semantic Versioning)
--------------------------------------------------------

The project uses the scheme: vX.Y.Z

  X = MAJOR  → Large restructurings, architecture changes (e.g., v2.0.0)
  Y = MINOR  → New features, new components (e.g., v2.1.0)
  Z = PATCH  → Bug fixes, CSS adjustments, warnings, typos (e.g., v2.0.1)

CURRENT VERSION: v2.0.8 (defined in instrucoes.txt)

INCREMENT RULES:
  - Fixed a CSS warning?                     → PATCH  (+0.0.1)
  - Added a new component/section?           → MINOR  (+0.1.0)
  - Refactored project architecture?         → MAJOR  (+1.0.0)

NEVER skip versions. Always read the current version in instrucoes.txt before
defining the next number.

--------------------------------------------------------
 SECTION 2 — UPDATE PROTOCOL (Step by Step)
--------------------------------------------------------

After ANY change in the project files, you MUST execute
the steps below, IN THE INDICATED ORDER:

STEP 1 — Determine the type of change and the new version
  → Classify the change: PATCH / MINOR / MAJOR
  → Calculate the new version number based on the current version of instrucoes.txt

STEP 2 — Update the documentation
  a) Edit `instrucoes.txt`:
     - Change "Current version: vX.Y.Z" to the new number
     - If MINOR or MAJOR, update "Next new feature" and "Major restructurings"

  b) Edit `README.md`:
     - Update the badge or version mention at the top of the file (if it exists)
     - Add an entry in the internal changelog (if the section exists)

STEP 3 — Execute git add (include all modified files)
  Command: git add .
  OR to be more precise: git add <file1> <file2> ...

STEP 4 — Execute git commit with standard message
  Mandatory commit message format:
    [TYPE] vX.Y.Z — Concise description of the change

  Accepted types:
    fix      → Bug or warning fix
    feat     → New feature
    refactor → Refactoring without behavior change
    style    → Visual change, CSS, animations
    docs     → Documentation update only
    test     → Adding/updating tests
    ci       → Change in CI/CD (GitHub Actions, deploy.yml)
    chore    → Maintenance tasks (deps update, etc.)

  Examples:
    git commit -m "style v2.0.1 — Adjust Aurora background wave animation"
    git commit -m "feat v2.1.0 — Add new internal agents dashboard section"

STEP 5 — Execute git push
  Command: git push origin main
  (GitHub Actions CI/CD will automatically deploy to GitHub Pages)

--------------------------------------------------------
 SECTION 3 — POST-PUSH VERIFICATIONS
--------------------------------------------------------

After push, confirm:
  1. GitHub Actions executed successfully (no errors in deploy.yml)
  2. Jest tests passed (npm run test)
  3. Production build compiles without errors (npm run build)

If any step fails:
  - Analyze the error BEFORE trying to fix
  - Make a new fix and repeat the protocol from STEP 1 to STEP 5
  - DO NOT force push (git push --force) without explicit maintainer approval

 SECTION 4 — GENERAL PROJECT RULES
--------------------------------------------------------

TECHNOLOGICAL STACK:
  - Framework:  Vite + React 19 + TypeScript
  - Style:      Tailwind CSS + Advanced Native CSS
  - Animations: Exclusive Native CSS (No Framer Motion, extreme performance)
  - Layout:     SPA with Horizontal Slides Transitions via pure State and Scroll Mouse Event
  - Deploy:     GitHub Actions → GitHub Pages (domain: aao.sh)

COLOR PALETTE (NEVER CHANGE WITHOUT APPROVAL):
  - Exclusive Focus on Dark Mode (Absolute blacks and Liquid Glass)

STRUCTURE FORBIDDEN TO CHANGE:
  - The horizontal SPA transition logic in `App.tsx`
  - The `React.memo` optimization of graphical components (`DotField`, `Aurora`, Sections)

STRUCTURE ALLOWED TO EXPAND:
  - New navigation tabs in the `TABS` array
  - Content insertion within Sections (`Hero`, `HowItWorks`, `Repo`)
  - Fine customization of Aurora in `index.css`
