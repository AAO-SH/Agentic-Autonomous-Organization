LEIA ESTE ARQUIVO ANTES DE QUALQUER AÇÃO NO REPOSITÓRIO.

Este documento define o protocolo que TODO agente de IA (LLM, Copilot, etc.)
deve seguir ao interagir com este repositório. O não cumprimento pode gerar
versões inconsistentes, histórico de commits sujo e falha no CI/CD.

--------------------------------------------------------
 SEÇÃO 1 — PADRÃO DE VERSIONAMENTO (Semantic Versioning)
--------------------------------------------------------

O projeto usa o esquema: vX.Y.Z

  X = MAJOR  → Grandes reestruturações, mudanças de arquitetura (ex: v2.0.0)
  Y = MINOR  → Novas features, novos componentes (ex: v2.1.0)
  Z = PATCH  → Correções de bugs, ajustes de CSS, warnings, typos (ex: v2.0.1)

VERSÃO ATUAL: v2.0.3  (definida no instrucoes.txt)

REGRAS DE INCREMENTO:
  - Corrigi um warning de CSS?               → PATCH  (+0.0.1)
  - Adicionei um novo componente/seção?      → MINOR  (+0.1.0)
  - Refatorei a arquitetura do projeto?      → MAJOR  (+1.0.0)

NUNCA pule versões. Sempre leia a versão atual no instrucoes.txt antes de
definir o próximo número.

--------------------------------------------------------
 SEÇÃO 2 — PROTOCOLO DE ATUALIZAÇÃO (Passo a Passo)
--------------------------------------------------------

Após QUALQUER alteração nos arquivos do projeto, execute OBRIGATORIAMENTE
os passos abaixo, NA ORDEM INDICADA:

PASSO 1 — Determinar o tipo de mudança e a nova versão
  → Classifique a mudança: PATCH / MINOR / MAJOR
  → Calcule o novo número de versão com base na versão atual do instrucoes.txt

PASSO 2 — Atualizar a documentação
  a) Editar `instrucoes.txt`:
     - Altere "Versão atual: vX.Y.Z" para o novo número
     - Se for MINOR ou MAJOR, atualize "Próxima feature nova" e "Grandes reestruturações"

  b) Editar `README.md`:
     - Atualize o badge ou menção de versão no topo do arquivo (se existir)
     - Adicione entrada no changelog interno (se existir seção)

PASSO 3 — Executar git add (incluir todos os arquivos modificados)
  Comando: git add .
  OU para ser mais preciso: git add <arquivo1> <arquivo2> ...

PASSO 4 — Executar git commit com mensagem padronizada
  Formato obrigatório da mensagem de commit:
    [TIPO] vX.Y.Z — Descrição concisa da mudança

  Tipos aceitos:
    fix      → Correção de bug ou warning
    feat     → Nova funcionalidade
    refactor → Refatoração sem mudança de comportamento
    style    → Mudança visual, CSS, animações
    docs     → Atualização de documentação apenas
    test     → Adição/atualização de testes
    ci       → Mudança em CI/CD (GitHub Actions, deploy.yml)
    chore    → Tarefas de manutenção (atualização de deps, etc.)

  Exemplos:
    git commit -m "style v2.0.1 — Adjust Aurora background wave animation"
    git commit -m "feat v2.1.0 — Add new internal agents dashboard section"

PASSO 5 — Executar git push
  Comando: git push origin main
  (O CI/CD do GitHub Actions fará o deploy automático para o GitHub Pages)

--------------------------------------------------------
 SEÇÃO 3 — VERIFICAÇÕES PÓS-PUSH
--------------------------------------------------------

Após o push, confirme:
  1. O GitHub Actions executou com sucesso (sem erros no deploy.yml)
  2. Os testes Jest passaram (npm run test)
  3. O build de produção compila sem erros (npm run build)

Se qualquer etapa falhar:
  - Analise o erro ANTES de tentar corrigir
  - Faça uma nova correção e repita o protocolo do PASSO 1 ao PASSO 5
  - NÃO force push (git push --force) sem aprovação explícita do mantenedor

--------------------------------------------------------
 SEÇÃO 4 — REGRAS GERAIS DO PROJETO
--------------------------------------------------------

STACK TECNOLÓGICA:
  - Framework:  Vite + React 19 + TypeScript
  - Estilo:     Tailwind CSS + CSS Nativo Avançado
  - Animações:  CSS Nativo exclusivo (Sem Framer Motion, performance extrema)
  - Layout:     SPA com Transições em Slides Horizontais via State puro e Scroll Mouse Event
  - Deploy:     GitHub Actions → GitHub Pages (domínio: aao.sh)

PALETA DE CORES (NUNCA ALTERAR SEM APROVAÇÃO):
  - Foco Exclusivo em Dark Mode (Pretos absolutos e Liquid Glass)

ESTRUTURA PROIBIDA DE ALTERAR:
  - A lógica de transição horizontal de SPA no `App.tsx`
  - A otimização `React.memo` dos componentes gráficos (`DotField`, `Aurora`, Sections)

ESTRUTURA PERMITIDA DE EXPANDIR:
  - Novas abas de navegação no `TABS` array
  - Inserção de conteúdo dentro das Sections (`Hero`, `HowItWorks`, `Repo`)
  - Personalização fina da Aurora no `index.css`

--------------------------------------------------------
 SEÇÃO 5 — EXEMPLO COMPLETO DE EXECUÇÃO
--------------------------------------------------------

Cenário: Você ajustou a opacidade da Aurora no index.css.

  1. Tipo: PATCH → nova versão: v2.0.1
  2. Atualize instrucoes.txt: "Versão atual: v2.0.1"
  3. git add src/index.css docs/instrucoes.txt
  4. git commit -m "style v2.0.1 — Update Aurora opacity for better contrast"
  5. git push origin main
  6. Verifique: Actions passou? Testes passaram? Build OK?
